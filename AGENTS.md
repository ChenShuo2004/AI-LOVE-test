# WARMTH 项目级规则

## 项目定位

WARMTH 是一个有温度的关系复盘产品。

当前 MVP 是「情侣关系复盘小游戏」：

- 单人测试：帮助用户先看清自己的感受。
- 双人测试：帮助双方看见差异，而不是争对错。
- 结果表达：关系天气、关系称号、今日复盘卡、给 TA 的一句话。

目标不是做严肃心理诊断，而是做一个温柔、可分享、低压力的关系沟通入口。

## 品牌规则

商标资产：

- 主商标：`public/brand/warmth-logo.png`
- 品牌名：`WARMTH`
- 中文副标：`有温度阅览室`

视觉关键词：

- 温柔
- 克制
- 安静
- 书卷感
- 陪伴感
- 留白
- 柔和光感

避免：

- 土味恋爱测试
- 过度粉色
- 强刺激情绪文案
- 审判式问题
- 花哨动效
- 夸张渐变

颜色方向：

- 背景：暖白、浅米、低饱和雾蓝
- 文本：墨黑、灰褐
- 点缀：柔和珊瑚、低饱和蓝灰

## 产品规则

所有问题和结果都要遵守：

1. 不评判谁对谁错。
2. 不制造焦虑。
3. 不诱导分手。
4. 不冒充心理咨询或医疗建议。
5. 多使用「感受、期待、修复、靠近」这类表达。
6. 输出要让用户愿意截图、愿意发给对方。

默认优先级：

1. 完整体验闭环。
2. 氛围感。
3. 分享传播。
4. 数据持久化。
5. AI 深度分析。

## 技术规则

当前技术栈：

- Vite
- React
- TypeScript
- CSS
- lucide-react

短期不要引入：

- 重型 UI 框架
- 复杂状态管理
- 后端服务
- 登录系统
- 支付系统

代码原则：

- 能放在一个页面完成的 MVP，不拆复杂架构。
- 问题、选项、结果文案必须结构化。
- 新功能先做可体验版本，再考虑抽象。
- 不把心理评估写成硬编码判断，文案要保留温柔余地。

## 环境规则

本地开发：

```bash
npm.cmd install
npm.cmd run dev
```

本地构建：

```bash
npm.cmd run build
```

环境变量：

- 示例文件：`.env.example`
- 本地文件：`.env.local`
- 任何真实密钥都只放 `.env.local`，不要提交到 Git。

## Git 规则

默认分支：`main`

提交格式使用 Conventional Commits：

- `feat:` 新功能
- `fix:` Bug 修复
- `refactor:` 重构
- `docs:` 文档
- `style:` 样式或格式
- `test:` 测试
- `chore:` 工程配置

每完成一个可运行版本，都要：

1. 跑 `npm.cmd run build`
2. 确认 `git status`
3. 提交一个清晰 commit

不要提交：

- `node_modules`
- `dist`
- `.env.local`
- TypeScript build cache
